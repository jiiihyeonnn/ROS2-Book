### 슬라이드 3: 복수 노드 통신의 문제점

**슬라이드 내용**

### 슬라이드 4: ROS 1 Nodelets vs ROS 2 IPC

**슬라이드 내용**
- ROS 1 Nodelets: 별도 API 필요, C++ 전용, zero-copy 제공
- ROS 2 IPC: 일반 Node API와 유사, rclcpp만 지원, zero-copy 구현
- 공통점: 단일 프로세스 내 복수 노드, 고정 메모리 공유
- ROS 2 IPC가 더 쉬운 API로 동일한 성능을 제공

**시각적 제안**
- 레이아웃: 2열 비교 테이블
- 시각 요소: ROS 1(회색) vs ROS 2(Accent) 대비

### 슬라이드 5: IPC의 핵심 요소

**슬라이드 내용**
- `use_intra_process_comms(true)` — 노드 생성 시 IPC 활성화
- `unique_ptr` 메시지 — 단일 소유권, 복사 없이 포인터만 이전
- `std::move` — 소유권을 퍼블리셔로 이전
- `SingleThreadedExecutor` — 복수 노드를 단일 프로세스에서 실행
- `shared_ptr` / `weak_ptr` — 안전한 공유·유효성 확인

**시각적 제안**
- 레이아웃: 5개 핵심 요소 카드
- 시각 요소: 각 요소에 아이콘 + 간결한 설명
- 강조: `unique_ptr` + `std::move` 조합을 Accent 컬러로 표현

### 슬라이드 6: 데모 1 — Two Node Pipeline

**슬라이드 내용**
- Producer 노드: 1초마다 Int32 메시지 생성 후 `std::move`로 발행
- Consumer 노드: 메시지 수신 후 메모리 주소 출력
- 두 노드를 동일한 `SingleThreadedExecutor`에 추가
- **결과**: 발행·수신 메시지의 메모리 주소가 동일 → zero-copy 확인

**시각적 제안**
- 레이아웃: 상단 코드 핵심 + 하단 실행 결과 터미널
- 시각 요소: Producer → Consumer 화살표 + 동일 주소 강조
- 강조: 동일한 메모리 주소를 Accent 컬러로 하이라이트

---
### 슬라이드 7: 데모 2 — Cyclic Pipeline

**슬라이드 내용**
- pipe1 (topic1 구독 → topic2 발행) + pipe2 (topic2 구독 → topic1 발행)
- 처음 한 번만 메시지 생성, 이후 **동일 메모리 객체가 두 노드 사이를 순환**
- 1초마다 값 증가, 메모리 주소는 변하지 않음
- 하나의 메시지 객체가 메모리 복사 없이 순환함을 확인

**시각적 제안**
- 레이아웃: 순환 구조 다이어그램 + 우측 실행 결과
- 시각 요소: pipe1 ↔ pipe2 순환 화살표
- 강조: 동일 메모리 주소를 Sub Accent 컬러로 표현

---
### 슬라이드 8: 데모 3 — Image Pipeline

**슬라이드 내용**
- `camera_node` → `watermark_node` → `image_view_node` 3단계 파이프라인
- 세 노드 모두 IPC 활성화, 단일 프로세스에서 실행
- 이미지에 PID와 메모리 주소가 표시 → 세 노드가 동일 주소 공유 확인
- 별도 프로세스로 실행한 `image_view_node`는 다른 PID·주소 → 메모리 복사 발생

**시각적 제안**
- 레이아웃: 3노드 파이프라인 다이어그램 + IPC 사용/미사용 비교 표
- 시각 요소: 단일 프로세스 박스 vs 복수 프로세스 박스
- 강조: 메모리 복사 발생 여부를 색상으로 대비

### 슬라이드 10: 핵심 정리
